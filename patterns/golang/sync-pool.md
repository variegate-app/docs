# sync.Pool / object pooling

## [<<< ---](../index.md)

`sync.Pool` — это механизм переиспользования объектов, уменьшающий давление на GC.

Идея object pooling:

- вместо создания нового объекта на каждую операцию;
- мы берем объект из пула, используем, приводим к “чистому” состоянию и возвращаем обратно.

```go
package main

import (
	"bytes"
	"fmt"
	"sync"
)

var bufPool = sync.Pool{
	New: func() any {
		return new(bytes.Buffer)
	},
}

func main() {
	buf := bufPool.Get().(*bytes.Buffer)
	// обязательно сбрасываем состояние объекта перед повторным использованием
	buf.Reset()

	buf.WriteString("hello")
	fmt.Println(buf.String())

	// возвращаем объект в пул
	bufPool.Put(buf)
}
```

Важно: `sync.Pool` — не гарантия “строгого” контроля объектов. Планировщик/GC может очищать пул по своему усмотрению, поэтому объект после `Get()` всегда должен быть корректно инициализирован/сброшен.

## Типовые ошибки / антипаттерны
- Не сбрасывать состояние объекта перед повторным использованием (`Reset`/обнуление).
- Хранить ссылки на объекты пула извне и использовать их после повторного помещения в `Put`.
- Полагаться на детерминированность (ожидая, что GC “точно” не тронет pool).

## Практический чеклист
- Объект после `Get()` всегда возвращается в “чистое” состояние до использования.
- После `Put()` нет внешних ссылок на объект (или ссылки не используются).
- В пул кладётся только то, что действительно выгодно переиспользовать (горячие аллокации).

## Как адаптировать под кейс
- Если критичны порядок/время жизни — `sync.Pool` может быть неподходящим, используйте явные менеджеры ресурсов.
- Для больших структур и частого использования проверьте эффект через `pprof`/метрики аллокаций.
