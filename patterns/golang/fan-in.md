# Fan-In

## [<<< ---](../index.md)

Fan-In — это паттерн обмена сообщениями, который создаёт «воронку» для работы нескольких источников (клиенты: source, сервер: destination).

Такое поведение удобно моделировать с помощью каналов Go.

```go
// Объединяет несколько входных каналов в один выходной
func Merge(cs ...<-chan int) <-chan int {
    var wg sync.WaitGroup

    out := make(chan int)

    // Для каждого входного канала из cs запускается горутина send.
    // Она копирует значения из c в out, пока c не будет закрыт, затем вызывает wg.Done.
    send := func(c <-chan int) {
        for n := range c {
            out <- n
        }
        wg.Done()
    }

    wg.Add(len(cs))
    for _, c := range cs {
        go send(c)
    }

    // Отдельная горутина закроет out, когда все send‑горутины завершатся.
    // Её нужно запускать только после вызова wg.Add.
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}
```

Функция `Merge` преобразует список каналов в один канал, создавая для каждого входного канала отдельную горутину, которая копирует его значения в общий выходной канал.

Когда все отправляющие горутины отработают, дополнительная горутина закрывает основной выходной канал.

## Типовые ошибки / антипаттерны
- Не дожидаться завершения “send”-горутин и не закрывать `out`: consumer будет ждать закрытия бесконечно.
- Пропустить закрытие одного из входных каналов: `Merge` продолжит ждать.
- Закрывать `out` из нескольких мест (send-after-close).

## Практический чеклист
- Для каждого входного канала есть отдельная goroutine, которая копирует значения в `out`.
- `wg.Add` делается до запуска goroutine.
- `out` закрывается ровно один раз после `wg.Wait()`.
- Все входные каналы корректно закрываются снаружи.

## Как адаптировать под кейс
- Если нужна реакция на отмену — добавьте `done`/`ctx` в отправочные горутины.
- Для сохранения порядка используйте отдельную стратегию (fan-in сам по себе порядок не гарантирует).
