# errgroup

## [<<< ---](../index.md)

`errgroup` — это способ запускать несколько конкурентных задач и:

- собрать первую/единственную ошибку;
- отменять остальные задачи при первой ошибке (через общий `context`).

Обычно используется вместо “ручного” `WaitGroup` + канал ошибок.

```go
package main

import (
	"context"
	"errors"
	"fmt"
	"time"

	"golang.org/x/sync/errgroup"
)

func main() {
	ctx := context.Background()

	g, ctx := errgroup.WithContext(ctx)

	g.Go(func() error {
		time.Sleep(100 * time.Millisecond)
		return errors.New("first task failed")
	})

	g.Go(func() error {
		// Вторая задача обязана слушать ctx.Done(), чтобы корректно прерваться.
		select {
		case <-ctx.Done():
			return ctx.Err()
		case <-time.After(2 * time.Second):
			return nil
		}
	})

	if err := g.Wait(); err != nil {
		fmt.Println("group error:", err)
	}
}
```

Ключевая идея: “первая ошибка” переводит `context` в отменённое состояние, а задачи, которые его слушают, должны остановиться.

## Типовые ошибки / антипаттерны
- Запускать задачи в `errgroup`, но внутри них не слушать `ctx.Done()` — тогда отмена не остановит работу.
- Использовать `errgroup` вместе с “ручным” `WaitGroup` так, что можно дважды ждать/закрывать/обрабатывать результаты.
- Проглатывать ошибки задач: теряется смысл “первая ошибка” как механизма прекращения.

## Практический чеклист
- `errgroup.WithContext(ctx)` используется, чтобы получить общий отменяемый `context`.
- Каждая задача возвращает ошибку корректно (или `nil` при успехе).
- Задачи, которые можно отменить, реально прекращают работу по `ctx.Done()`.
- После `g.Wait()` ошибка обрабатывается как “причина отмены” или итог процесса.

## Как адаптировать под кейс
- Для операций с внешними ресурсами передавайте `ctx` внутрь (DB/RPC/HTTP), чтобы отмена работала на нижнем уровне.
- Если нужна стратегия “собрать все ошибки”, используйте другой подход (а не errgroup с отменой по первой ошибке).
