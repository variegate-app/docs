# Explicit cancellation

## [<<< ---](../index.md)

Паттерн **Explicit cancellation** подчёркивает явную, а не неявную отмену работы горутин: вызывающий код в явном виде посылает сигнал отмены, а все участники конвейера обязаны его обрабатывать.

В Go это реализуется через:

- отдельный канал `done`;
- или `context.Context`.

```go
func worker(done <-chan struct{}, jobs <-chan int, results chan<- int) {
	for {
		select {
		case <-done:
			// получен явный сигнал отмены
			return
		case j, ok := <-jobs:
			if !ok {
				return
			}
			results <- j * 2
		}
	}
}

func main() {
	done := make(chan struct{})
	jobs := make(chan int)
	results := make(chan int)

	go worker(done, jobs, results)

	// отправляем несколько задач
	for i := 0; i < 3; i++ {
		jobs <- i
	}

	// явно отменяем дальнейшую работу
	close(done)
	close(jobs)
}
```

Явная отмена позволяет избежать утечек горутин и «подвисания» каналов.

## Типовые ошибки / антипаттерны
- Закрыть `done` раньше, чем запущенные горутины успеют “подписаться” на сигнал (возможны гонки логики, хотя `close(done)` сам по себе безопасен).
- Не закрывать/не освобождать входные каналы (`jobs`), из-за чего worker может ждать “следующую задачу” дольше, чем нужно.
- Отправлять результаты в канал без consumer при отмене (получаются блокировки/утечки).

## Практический чеклист
- У каждого worker есть select-ветка на `done` и корректный `return`/`break`.
- Входной канал завершится (обычно через `close(jobs)` у координатора).
- Любой канал результатов или дренится, или не используется в сценарии отмены.
- Порядок закрытия: сначала отмена (при необходимости), затем закрытие входов/координация.

## Как адаптировать под кейс
- Если `done` используется как “глобальная отмена” для пачки задач — сделайте его параметром координатора.
- Если нужна отмена с дедлайном — переходите на `context.WithTimeout`.
