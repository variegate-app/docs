# Steady-State

## [<<< ---](../index.md)

Паттерн **Steady-State** описывает систему, которая после фазы разогрева выходит на устойчивый режим работы: количество входящих задач примерно равно количеству обрабатываемых, а пулы воркеров и буферы подобраны так, чтобы очередь не росла бесконечно.

Обычно это комбинация:

- ограниченного пула воркеров;
- буферизованной очереди заданий;
- мониторинга длины очереди и времени обработки.

```go
type Job struct {
	ID int
}

func worker(done <-chan struct{}, jobs <-chan Job) {
	for {
		select {
		case <-done:
			return
		case j, ok := <-jobs:
			if !ok {
				return
			}
			// имитация работы
			time.Sleep(50 * time.Millisecond)
			fmt.Println("обработана задача", j.ID)
		}
	}
}

func main() {
	done := make(chan struct{})
	defer close(done)

	jobs := make(chan Job, 100) // очередь заданий

	// фиксированное число воркеров
	for i := 0; i < 4; i++ {
		go worker(done, jobs)
	}

	// постоянный приток задач с контролируемой скоростью
	for id := 0; id < 1000; id++ {
		jobs <- Job{ID: id}
	}
	close(jobs)
}
```

Подбирая размер пула и буфера, можно добиться устойчивого режима без раздувания очередей и истощения ресурсов.

## Типовые ошибки / антипаттерны
- Игнорировать мониторинг длины очереди: steady-state “съезжает”, когда нагрузка меняется.
- Делать буфер бесконечным: система перестаёт быть предсказуемой и накапливает задержку.
- Запускать слишком много воркеров “на всякий случай”: ухудшается latency и растёт contention.

## Практический чеклист
- Есть ограничение параллелизма (воркер пул) и ограничение буфера очереди.
- Есть измерения: длина очереди, время обработки, время ожидания.
- При росте нагрузки включается политика деградации (например, rate limit/drop) или расширение.
- Есть понятная остановка (stop: закрытие очереди + ожидание воркеров).

## Как адаптировать под кейс
- Steady-state обычно достигается сочетанием `worker pool` + `bounded queue` + backpressure.
