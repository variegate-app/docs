# Stopping short

## [<<< ---](../index.md)

Паттерн **Stopping short** описывает ситуацию, когда потребитель потока данных принимает решение завершить работу раньше, чем исчерпан весь входной канал, и должен корректно остановить остальных участников конвейера.

Обычно это достигается с помощью канала `done` или `context.Context`, который слушают все стадии конвейера.

```go
// generator генерирует бесконечную последовательность чисел.
func generator(done <-chan struct{}) <-chan int {
	out := make(chan int)
	go func() {
		defer close(out)
		for i := 0; ; i++ {
			select {
			case <-done:
				return
			case out <- i:
			}
		}
	}()
	return out
}

func main() {
	done := make(chan struct{})
	defer close(done)

	stream := generator(done)

	// берём только первые 5 значений и останавливаемся
	for i := 0; i < 5; i++ {
		fmt.Println(<-stream)
	}

	// закрытие done приведёт к остановке генератора
}
```

Без явного сигнала остановки генератор продолжал бы работать и после того, как потребитель перестал читать из канала, что привело бы к утечке горутины.

## Типовые ошибки / антипаттерны
- Остановиться “снаружи”, но не остановить продюсеров: горутины продолжают работать и утекут.
- Не закрыть выходной канал источника при отмене: pipeline зависнет.
- Игнорировать `done`/`ctx.Done()` в циклах отправки.

## Практический чеклист
- Есть общий `done`/`ctx` который слушают все участники конвейера.
- Когда потребитель решает завершиться — сигнал отмены подаётся координатором.
- Источник/стадии корректно выходят по отмене и закрывают свои output'ы.
- Нет зависаний на отправках, когда дальше никто не читает (используйте select с `done`).

## Как адаптировать под кейс
- Для “взять первые N и остановиться” используйте `take-first-n`.
- Если остановка нужна по событию ошибки — комбинируйте с fail-fast/cancellation.
