# Wait For Result

## [<<< ---](../index.md)

Основная идея паттерна **Wait For Result** — разделить выполнение работы и ожидание её результата:

- есть канал, который используется как канал сигналов;
- есть горутина, которая делает работу;
- есть горутина, которая ждёт завершения этой работы.

### Пример

В этом примере есть `employee` (горутина `a`), у которого есть работа, и `manager` (горутина `main`), который ждёт результата. Когда работа выполнена, `employee` уведомляет `manager`, посылая сигнал (`paper`) через канал `ch`.

```go
package main

func main() {
    // создаём канал типа string, который даёт семантику сигналов
    ch := make(chan string)

    // горутина "a", которая делает работу => employee
    go func() {
        // имитируем непредсказуемую задержку (в проде так делать не стоит)
        time.Sleep(time.Duration(rand.Intn(500)) * time.Millisecond)
        // по завершении работы отправляем сигнал
        ch <- "paper"

        // мы не знаем, в каком порядке выполнятся print‑выражения
        fmt.Println("employee: sent signal")
    }()

    // горутина "main" => manager
    // горутины "main" и "a" выполняются параллельно

    // ждём и получаем сигнал от горутины "a"
    // блокирующая операция
    p := <-ch

    // мы не знаем, какой println выполнится раньше
    fmt.Println("manager: received signal :", p)

    // просто даём достаточно времени, чтобы увидеть результат (в проде так не делаем)
    time.Sleep(time.Second)
}

```

### Результат (1‑й запуск)

```go
go run main.go
employee: sent signal
manager: received signal : paper
```

### Результат (2‑й запуск)

```
go run main.go
manager: received signal : paper
employee: sent signal

```

## Частые заблуждения

1. Обрати внимание: в первом и втором запуске порядок выполнения вызовов `fmt.Println()` не гарантирован. Горутины выполняются параллельно.

## Типовые ошибки / антипаттерны
- Ждать результат без канала/контроля завершения: если worker упадёт/зависнет — wait будет бесконечным.
- Пытаться предполагать порядок событий при параллельном выполнении.
- Не использовать буфер/каналы корректно: возможна блокировка при отправке результата.

## Практический чеклист
- Канал сигналов/результатов имеет гарантированный sender и receiver.
- Для ожидания предусмотрена отмена/таймаут (особенно при ожидании внешней работы).
- Ошибки и отмена передаются тем же механизмом завершения, а не “теряются”.
- Порядок выводов не предполагается — проектируйте код без race на логике.

## Как адаптировать под кейс
- Для задач с пулом используйте wait-for-task в связке с worker pool.
- Для RPC/запросов используйте round-tripper или контекст поверх ожидания.
