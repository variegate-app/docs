# Fault-tolerance

## [<<< ---](../index.md)

Паттерн **Fault-tolerance** (отказоустойчивость) в контексте горутин и каналов обычно означает:

- изоляцию ошибок отдельных воркеров;
- автоматический рестарт упавших горутин;
- продолжение обработки, даже если часть компонентов вышла из строя.

Простейший вариант — «надзиратель» (supervisor), который перезапускает воркера при панике.

```go
func supervise(done <-chan struct{}, worker func(<-chan struct{})) {
	for {
		select {
		case <-done:
			return
		default:
		}

		// запускаем воркера с защитой от паники
		func() {
			defer func() {
				if r := recover(); r != nil {
					fmt.Println("worker crashed, restarting:", r)
				}
			}()

			worker(done)
		}()
	}
}
```

В реальных системах отказоустойчивость также дополняется лимитом на число перезапусков и экспоненциальной задержкой между рестартами.

## Типовые ошибки / антипаттерны
- Перезапускать бесконечно без лимита: “аварийный шторм” и деградация всей системы.
- Игнорировать очистку ресурсов после паники: утечки файлов/соединений/горутин.
- Не ограничивать частоту рестартов и не добавлять backoff/jitter.

## Практический чеклист
- Supervisor ограничивает число перезапусков и использует backoff между попытками.
- При перезапуске состояние воркера инициализируется заново (без накопления мусора).
- Ошибки/паники логируются и коррелируются с метриками.
- Внутренние горутины корректно завершаются при `done/ctx.Done()` и при остановке supervisor.

## Как адаптировать под кейс
- Для сервисов используйте circuit breaker и/или fallback, чтобы уменьшать нагрузку на нестабильный компонент.
