# Worker Pool

## [<<< ---](../index.md)

Паттерн **Worker Pool** описывает набор одинаковых worker‑горутин, считывающих задачи из общего канала и выполняющих их параллельно.

Он полезен, когда:

- есть большое количество однотипных задач;
- мы хотим ограничить максимальное число параллельных исполнителей.

```go
type Task func() error

// StartWorkerPool запускает poolSize воркеров, читающих задачи из tasks.
// Ошибки, если нужны, можно отправлять в отдельный канал.
func StartWorkerPool(done <-chan struct{}, poolSize int, tasks <-chan Task) {
	var wg sync.WaitGroup
	wg.Add(poolSize)

	for i := 0; i < poolSize; i++ {
		go func(id int) {
			defer wg.Done()
			for {
				select {
				case <-done:
					return
				case t, ok := <-tasks:
					if !ok {
						return
					}
					_ = t() // обработку ошибки можно добавить при необходимости
				}
			}
		}(i)
	}

	go func() {
		wg.Wait()
		// здесь можно закрыть канал результатов, если он есть
	}()
}
```

Такой пул можно переиспользовать для обработки запросов, задач из очереди, файлов и т.д.

## Типовые ошибки / антипаттерны
- Создавать workers без понятного механизма остановки (нет ветки на `done` и/или канал задач не закрывается).
- Не закрывать входной канал `tasks`: workers будут ждать “следующую задачу” и не завершатся.
- Игнорировать дренирование результатов/ошибок (если они вынесены в отдельные каналы).

## Практический чеклист
- Есть координатор, который гарантирует завершение: закрывает `tasks` и/или закрывает `done`.
- У каждого worker есть выход из бесконечного цикла по `done` и по закрытию `tasks`.
- Параллелизм ограничен `poolSize` (нет неконтролируемого размножения горутин).
- Ошибки либо обрабатываются внутри задачи, либо идут в отдельный канал с корректной утилизацией.
- `WaitGroup` используется для корректного завершения и/или закрытия выходных каналов.

## Как адаптировать под кейс
- Если задачи имеют разное время выполнения, worker pool обычно стабилен — но проверьте требования к порядку результатов.
- Для backpressure связывайте pool с буфером/очередью: координатор не должен “засыпать” workers бесконечными задачами.
