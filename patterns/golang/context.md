# Context

## [<<< ---](../index.md)

Паттерн **Context** в Go реализован стандартным пакетом `context` и используется для:

- распространения сигналов отмены;
- задания дедлайнов и таймаутов;
- передачи сквозных метаданных (request id, пользователь и т.п.).

Контекст обычно передаётся первым аргументом в функции, выполняющие работу.

```go
func doWork(ctx context.Context, id int) error {
	select {
	case <-time.After(200 * time.Millisecond):
		// работа завершилась вовремя
		return nil
	case <-ctx.Done():
		// контекст отменён или сработал дедлайн
		return ctx.Err()
	}
}

func main() {
	// создаём контекст с таймаутом 100 мс
	ctx, cancel := context.WithTimeout(context.Background(), 100*time.Millisecond)
	defer cancel()

	if err := doWork(ctx, 1); err != nil {
		fmt.Println("work cancelled:", err)
	}
}
```

Все функции, работающие с внешними ресурсами (БД, сеть, RPC), должны принимать `context.Context`, чтобы корректно реагировать на отмену и дедлайны.

## Типовые ошибки / антипаттерны
- Игнорировать отмену: долгие операции не слушают `ctx.Done()` (и не прекращают работу вовремя).
- Не передавать `context.Context` дальше по цепочке вызовов (переходить на глобальные/статические “контексты”).
- Забывать освобождать ресурсы, связанные с контекстом (например, не вызывать `cancel()` для `WithCancel/WithTimeout/WithDeadline`).

## Практический чеклист
- `ctx` передаётся первым аргументом во все функции, которые выполняют работу с внешними ресурсами.
- В долгих циклах есть точка выхода по `ctx.Done()`.
- Для запросов/операций выставлен дедлайн или таймаут (где это уместно).
- В коде, использующем `context.WithCancel/WithTimeout`, `cancel()` вызывается по завершении работы.

## Как адаптировать под кейс
- Определите границу: кто “владеет” жизненным циклом `ctx` (обычно координатор/обработчик запроса).
- Для фоновых задач выбирайте `Background/TODO` аккуратно: лучше создавайте производный контекст с таймаутом.
